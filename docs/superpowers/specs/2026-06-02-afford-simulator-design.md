# Симулятор «Дотяну ли на покупку?» — дизайн

Дата: 2026-06-02. Файл: `index.html` (один файл, vanilla, без сборки).

## Зачем

На дашборде есть «сколько свободно в день», но это статичная цифра. Пользователь хочет интерактивную прикидку: «свободно 6к/день, хочу куртку за 20к — если буду тратить меньше, за сколько накоплю; или потратить сейчас». Чистый «что если» со связанными ползунками, без сохранения.

## Размещение

Кнопка под карточкой дневного бюджета на дашборде → открывает модалку (динамический host `afford-sim-modal`, как у оплаты/итогов). Закрытие = `remove()`. В `state` ничего не пишем — миграции и `ALLOWED_KEYS` не трогаем.

## Данные (авто из приложения)

- `B` = `getSpendingPicture().breakEvenDaily` — свободно в день;
- `remainingFree` = `getSpendingPicture().remainingFree` — остаток свободного на месяц (для блока «купить сейчас»);
- `daysLeft` = `daysLeftInMonth()` — успеешь ли в этом месяце.

Состояние модалки живёт только в DOM (поле цены + 2 ползунка), пересчёт на `oninput`.

## Математика — чистая функция `simAfford({B, P, S, N, driver})`

- `B` — свободно/день, `P` — цена покупки, `S` — трачу/день, `N` — дней до покупки.
- Излишек/день = `B − S`. Связь: `N = ⌈P / (B − S)⌉`, `S = max(0, B − P/N)`.
- `driver: 'spend'` — ведущий ползунок S, пересчитываем N. `driver: 'days'` — ведущий N, пересчитываем S.
- `Nmin = ⌈P / B⌉` (когда тратишь 0). `Nmax` — например 90.
- Возвращает `{S, N, perDaySaved: B−S, doneDate, feasibleThisMonth: N ≤ daysLeft, buyNowLeftDaily, buyNowShortfall}`.

## Интерфейс модалки

- Поле **Цена покупки** `P` (number, без названия покупки — разовая прикидка).
- Строка «Свободно в день: `B`».
- **Ползунок 1 — «Трачу в день»** `S`, 0…B.
- **Ползунок 2 — «Готово за»** `N` дней, Nmin…Nmax.
- Тянешь любой → второй пересчитывается через `simAfford`.
- Живые читалки: «Откладываю `(B−S)`/день», «Накоплю за `N` дней — к {дата}», ✓/⚠ относительно конца месяца.
- Блок **«или купить сейчас»**: `remainingFree ≥ P` → «бюджет станет `(remainingFree−P)/daysLeft`/день»; иначе «сейчас не хватает `P−remainingFree`».

## Краевые случаи

- `B ≤ 0` → текст «Свободных на день сейчас нет — сначала разберись с бюджетом», ползунки скрыты.
- `P` пусто/≤0 → читалки молчат («укажи цену»).
- `S = B` → «так не накопишь (откладываешь 0/день)».
- `N` ниже `Nmin` → клампим к `Nmin`.
- Деление на ноль/Infinity — отсекаем (B−S>0 гарантируем).

## Код

- `simAfford(...)` — чистая, тестируется отдельно (node-харнесс).
- `openAffordSim()` — строит модалку; `updateAffordSim(driver)` — пересчёт на `oninput`; `closeAffordSim()`.
- Кнопка в разметке дашборда под `daily-card`, `onclick="openAffordSim()"`.
- Цвета — только CSS-переменные (`--good`, `--accent2`, `--muted`).
- `CACHE_VERSION` bump.

## Дополнение 1 — накопительный эффект дисциплины (две точки показа)

Стабильная дневная норма: `dailyAllowance = monthlyFree / daysInMonth` (сколько в среднем можно/день, чтобы за месяц выйти в ноль по свободному). Реюз `getSpendingPicture()` (monthlyFree, spentThisMonth, remainingFree, avgFact, daysLeft) + `daysInMonth()`, `daysPassedInMonth()`.

**1a. Блок на дашборде** (у карточки дневного бюджета), показывается только при `monthlyFree > 0`:
- `streak` — число последних **завершённых** дней подряд (со вчера назад), где дневные импульсные траты (`isDailySpend`) ≤ `dailyAllowance`;
- `surplusToDate = dailyAllowance * daysPassed − spentThisMonth` — «сверх плана отложено +X» (минус → «перебор −X»);
- `projEndSurplus = remainingFree − avgFact * daysLeft` — «при нынешнем темпе к концу месяца: +Y свободных» (минус → перебор).
- Тексты: «🔥 N дней подряд ниже нормы · сверх плана +X · к концу месяца +Y». Стрик показываем при `streak ≥ 2`, иначе только две суммы.
- Чистая функция `getDisciplineSnapshot()` → `{dailyAllowance, streak, surplusToDate, projEndSurplus}`; рендер `renderDisciplineBlock()` в `renderV2`.

**1b. Строка в симуляторе** (под читалками): при текущем `S` — `endSurplus = remainingFree − S * daysLeft` → «если держать этот темп до конца месяца — свободным останется +Z» (минус → перебор). Пересчитывается тем же `updateAffordSim`.

## Дополнение 2 — разбивка расхода в Итогах месяца (плановые / ежедневные)

В `computeMonthlyReview` добавить:
- `plannedExpense` = сумма tx с `isPlannedTx` (аренда/подписки по графику);
- `dailyExpense` = сумма прочих расходных не-переводных tx;
- (инвариант: `plannedExpense + dailyExpense === expense`).

В `renderMonthlyReviewBody`: под KPI «Расход» — строка «плановые X · ежедневные Y». **Backward-compat:** старые замороженные объекты в `monthlyReviews` поля не имеют → если `plannedExpense` undefined, показываем только общий расход без разбивки (не падаем).

## Инварианты

- Числа берём из существующих `getSpendingPicture()` / `daysLeftInMonth()` / `daysInMonth()` — не пересчитываем бюджет заново.
- Тон читалок прямой: «не накопишь», «не хватает», «перебор», без «возможно».
- Ничего не пишем в `state` — симулятор и блок дисциплины read-only. Итоги месяца: новые поля считаются на лету для текущего и пишутся в новые замороженные объекты (старые не ломаем).
- Цвета — только CSS-переменные (`--good` сэкономил/норма, `--accent2` перебор, `--muted`).
